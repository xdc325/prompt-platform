# 测试套件功能 — 测试手册

## 功能简介

测试套件用于对提示词版本进行**回归测试**。你可以创建一组测试用例（输入 + 期望关键词），选择一个版本和模型，系统会在后台逐个执行测试用例，检查 LLM 输出是否包含期望关键词，并汇报通过率。

## 前置条件

1. 已注册并登录系统
2. 已创建项目和提示词
3. 已创建至少一个提示词版本（建议内容包含 `{input}` 占位符）
4. 已配置 DeepSeek API Key（当前 `.env` 中有 `DEEPSEEK_API_KEY`）

## 测试步骤

### 步骤 1：准备提示词和版本

1. 打开 http://localhost
2. 登录后，进入一个项目，点击一个提示词
3. 如果还没有版本，点击「+ 新建版本」，输入内容例如：

```
你是一个客服。用户说：{input}
请用礼貌、专业的语气回复。
```

4. 点击「创建草稿」

---

### 步骤 2：创建测试套件

1. 在版本管理下方，找到「测试套件」区域
2. 点击「+ 新建测试套件」
3. 填写：
   - **套件名称**：例如 `回归测试`
   - **测试用例**：每个用例包含两个字段
     - `输入`：替换提示词中 `{input}` 的文本，例如 `我要退货`
     - `期望包含`：输出中应该包含的关键词，例如 `退货`
   - 点击「+ 添加测试用例」可以增加更多用例
4. 示例用例：

| 输入 | 期望包含 | 说明 |
|------|---------|------|
| 我要退货 | 退货 | 测试退货场景 |
| 物流太慢了 | 抱歉 | 测试投诉场景 |
| 怎么退款 | 退款 | 测试退款咨询 |

5. 点击「创建套件」

**预期结果**：套件卡片出现在列表中，显示名称和用例数量，每个用例的输入和期望关键词可见。

---

### 步骤 3：运行测试

1. 在套件卡片的操作区，有两个下拉框：
   - **版本选择**：选择要测试的版本（如 v1 (草稿)）
   - **模型选择**：选择 `DeepSeek Chat`（默认）
2. 点击「运行测试」按钮
3. 按钮显示「运行中...」并禁用

**预期结果**：
- 等待约 3-5 秒后，套件卡片下方显示测试结果
- 显示通过率（如 `100%`）
- 每个用例显示 ✓ 通过 或 ✗ 失败
- 通过的用例背景为绿色，失败的为红色
- 每个用例显示实际 LLM 输出（截断到 200 字符）

---

### 步骤 4：查看测试结果详情

- ✓ 通过：表示 LLM 输出中包含了期望的关键词
- ✗ 失败：表示输出中没有包含期望关键词（或 LLM 调用出错）
- 如果有错误（如 API Key 无效），失败用例会显示具体错误信息

---

### 步骤 5：修改测试套件

1. 目前前端暂不支持直接编辑套件（可通过 API 更新）
2. 可以删除旧套件后重新创建

---

### 步骤 6：删除测试套件

1. 点击套件卡片右上角的「删除」按钮
2. 确认删除提示

**预期结果**：套件及其关联的测试运行记录被删除，列表刷新。

---

### 步骤 7：重复运行保护

1. 如果某个套件正在运行中（状态为 running），再次点击「运行测试」
2. 页面会提示「A test is already running for this suite」（冲突保护）

## API 测试（可选，用 curl 验证）

如果你习惯用命令行，以下是完整的 API 测试命令：

```bash
# 登录
TOKEN=$(curl -s -X POST http://localhost/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"你的邮箱","password":"你的密码"}' \
  | python -c "import sys,json; print(json.load(sys.stdin)['data']['access_token'])")

# 创建测试套件
curl -s -X POST "http://localhost/api/v1/prompts/{prompt_id}/test-suites" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $TOKEN" \
  -d '{
    "name": "API Test Suite",
    "test_cases": [
      {"input": "测试输入1", "expected": "期望关键词1"},
      {"input": "测试输入2", "expected": "期望关键词2"}
    ]
  }'

# 运行测试（指定模型）
curl -s -X POST "http://localhost/api/v1/prompts/{prompt_id}/test-suites/{suite_id}/run" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"version_id": "{version_id}", "model": "deepseek-chat"}'

# 查看测试结果
curl -s "http://localhost/api/v1/test-runs/{run_id}" \
  -H "Authorization: Bearer $TOKEN"

# 删除测试套件
curl -s -X DELETE "http://localhost/api/v1/prompts/{prompt_id}/test-suites/{suite_id}" \
  -H "Authorization: Bearer $TOKEN"
```

## 边界情况测试

| 测试场景 | 预期结果 |
|---------|---------|
| 套件名称为空 | 返回验证错误 |
| 测试用例为空 | 返回验证错误 |
| 用例 input 为空 | 返回验证错误 |
| 查询不存在的测试运行 | 返回 "TestRun not found" |
| 删除有关联测试运行的套件 | 级联删除，成功 |
| 同时运行同一套件两次 | 第二次返回 "A test is already running..." |
| 更新套件名称（保留用例） | 名称更新，用例不变 |
| 更新套件用例（保留名称） | 用例更新，名称不变 |
| 使用 deepseek-chat 模型 | 正常工作（当前已配置） |
| 使用 deepseek-reasoner 模型 | 应正常工作（需确认 API 支持） |
| 使用 gpt-3.5-turbo（无 Key） | 返回 API Key 未配置错误 |

## 已修复的问题

以下问题在本次测试中已发现并修复：

1. **arq 未注入** — 测试套件运行后 job 从未入队，状态永远是 running。修复：添加 `get_arq` 依赖注入。
2. **Worker 未 commit** — Worker 执行完测试后没有 `session.commit()`，结果未写入数据库。修复：添加 `await session.commit()`。
3. **datetime 时区冲突** — `datetime.now(timezone.utc)` 是 aware datetime，但数据库列是 `TIMESTAMP WITHOUT TIME ZONE`。修复：改用 `datetime.utcnow()`。
4. **Worker 模型硬编码** — Worker 之前写死 `gpt-3.5-turbo`，用户只有 DeepSeek Key。修复：添加 `model` 参数从前端传给 worker。
5. **输入渲染错误** — Worker 将 `case["input"]`（字符串）当 dict 遍历，导致字符级替换。修复：直接替换 `{input}` 占位符。

## 已知限制

- 提示词模板目前仅支持 `{input}` 占位符。其他变量（如 `{role}`、`{style}`）不会自动填充，会原样传给 LLM。
- 前端暂不支持直接编辑已创建的测试套件（需通过 API PUT 端点）。
- 测试套件的结果页面不会自动刷新，需手动点击运行查看新结果。
